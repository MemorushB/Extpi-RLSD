# ExtPI-RLSD：交给 Codex 的实现与快速验证方案

## 0. 项目目标与结论先行

本项目在同一套数学推理数据、Qwen3-1.7B student、LoRA 和 1×H200 约束下实现并比较：

1. `closed_sft`：闭源模型离线生成可见解题过程，Qwen3-1.7B 做 completion-only SFT；
2. `opd_pg_qwen8b`：Qwen3-1.7B 采样当前策略轨迹，Qwen3-8B 在同一 prefix 上对这些 token 打分，做直接 external on-policy distillation；
3. `extpi_rlsd_qwen8b_trace`：Qwen3-8B 离线生成经过答案验证的解题轨迹，把它作为 Qwen3-1.7B self-teacher 的 privileged information（PI），再用 RLSD 对 GRPO token advantage 做信用重分配；
4. 必须附带 `grpo` 和 `extpi_rlsd_shuffled` 两个诊断对照。

核心研究问题：

> Can verified reasoning traces generated by Qwen3-8B improve a Qwen3-1.7B PI-free policy through recipient-grounded RLSD credit assignment, approaching direct Qwen3-8B on-policy distillation while outperforming off-policy SFT?

本方法不是直接的 `Qwen3-8B logits → Qwen3-1.7B` 蒸馏。准确机制是：

```text
Qwen3-8B
   └─ offline verified reasoning trace z
         └─ Qwen3-1.7B self-teacher conditioned on z
               └─ token-level PI evidence Δ
                     └─ RLSD reweighted GRPO update
                           └─ PI-free Qwen3-1.7B policy
```

建议论文/实验命名：

- **ExtPI-RLSD**
- **External Privileged-Trace RLSD**
- **Recipient-Grounded Privileged-Trace Transfer**

禁止把它命名为 direct Qwen3-8B-to-1.7B logit distillation。

## 2. 代码底座选择

### 2.1 推荐策略

以**当前 upstream verl 的一个固定 commit**为底座，不从 RLSD 的 EasyVideoR1 fork 继续开发。

原因：

- RLSD 公开仓库主要是多模态 EasyVideoR1/FSDP 路径；
- 其核心 RLSD 算法本身很小，主要是 token advantage reweighting；
- 当前 verl 已有 LoRA/PEFT 和 OPD 相关基础设施；
- 单卡 H200 不适合官方“独立 teacher GPU resource pool”布局，需要在 upstream 上增加同卡串行 scorer backend；
- 这样 direct OPD 和 ExtPI-RLSD 可以共用同一个 scorer contract。

第一条 Codex 任务：

```bash
git clone <upstream verl>
git checkout <chosen pinned commit>
git switch -c extpi-rlsd-mvp
```

先运行 upstream 自带的 Qwen3-1.7B LoRA smoke test。未通过前不得加入 RLSD 改动。

### 2.2 最小侵入原则

优先复用 upstream 的：

- PPO/GRPO trainer loop；
- response mask；
- old logprob 计算；
- reward/advantage 计算；
- LoRA actor + vLLM weight sync；
- 官方 OPD loss/data contract（若该 commit 已包含）。

新增能力应是：

1. `inline_hf` teacher scorer；
2. privileged teacher prompt assembler；
3. RLSD advantage transformer；
4. recipe、数据生成和测试。

不要复制一整套 PPO trainer。

## 5. 先做 PI 可用性验证，再训练

这是整个项目最重要的 go/no-go gate。

### 5.1 Qwen3-8B PI 生成

对候选训练题，Qwen3-8B 只看到问题，不看到 gold solution。建议：

```yaml
thinking_mode: true
temperature: 0.6
top_p: 0.95
top_k: 20
max_new_tokens: 4096
max_attempts_per_prompt: 4
```

输出要求：

```text
- self-contained reasoning
- final answer in \boxed{...}
- no mention of gold/reference solution
```

每个 candidate 使用与训练完全相同的 `math_verify` 验证。选择**最短的 verified-correct trace**作为 PI。不得在失败时静默替换成数据集 gold solution。

保存全部 attempts，以便审计；主数据只引用所选 trace ID。

### 5.2 Recipient PI uplift

对每道有正确 8B trace 的题，用 frozen/base Qwen3-1.7B 各采样 4 次：

```text
plain: problem only
PI: problem + verified Qwen3-8B reference trace
```

计算：

```text
p_plain = correct_plain / 4
p_PI    = correct_PI / 4
uplift  = p_PI - p_plain
```

快速验证 frontier 的推荐条件：

```text
p_plain ∈ {0.25, 0.50, 0.75}
p_PI >= 0.75
uplift >= 0.25
PI trace verified
student completion not truncated
```

目标构建：

```text
mvp_train: 800 prompts
matched_dev: 512 prompts, disjoint
smoke: 64 prompts
```

如果不足 800 条，可放宽为：

```text
p_plain in [0.1, 0.75]
uplift > 0
```

但必须报告筛选规则和覆盖率。

### 5.3 无偏确认集

Frontier 是机制验证集，不是最终泛化结论。主机制成功后，再从剩余 clean OPSD 数据中固定抽取：

```text
confirmation_train: 3200 prompts
confirmation_dev: 512 prompts
```

不可按 PI uplift 过滤，仅要求 PI trace verified 和长度合格。

## 6. 公共 prompt 与 thinking 配置

### 6.1 主实验采用 non-thinking student

为节省 rollout 成本并减少 direct OPD 的风格错配：

```text
student rollout: Qwen3 non-thinking
Qwen3-8B direct OPD scorer: non-thinking prompt
Qwen3-1.7B PI scorer: non-thinking prompt
Qwen3-8B offline PI generation: thinking enabled
```

Primary student prompt：

```text

End with the final answer in \boxed{...}.

{problem}
```

Direct OPD teacher prompt必须与 student prompt 的 system/user/chat template 完全相同。

PI scorer prompt：

```text
Solve the following mathematics problem. Give a concise, self-contained derivation.

{problem}

</reference_solution>
```

注意：PI prompt 只用于 scorer；绝不能进入 student rollout context。

可追加：

```text

```

但不得把它混入主结果，因为 direct OPD baseline 也需要相应适配。

## 7. 方法定义

## 7.1 Baseline 1：closed-source off-policy SFT

闭源模型 API 使用 provider-agnostic adapter：

```yaml
provider: openai_compatible
base_url: env:CLOSED_TEACHER_BASE_URL
api_key: env:CLOSED_TEACHER_API_KEY
model: env:CLOSED_TEACHER_MODEL
```

不要请求或存储隐藏 chain-of-thought。请求“可见、简洁、自洽的 derivation + boxed final answer”。

每题最多重试 4 次；只有自动验证正确的输出进入 SFT。保存：

- raw response；
- provider/model/version；
- request ID；
- generation params；
- attempt index；
- verifier result；
- timestamp；
- prompt hash。

SFT：

```text
student input = common student prompt
label = closed teacher visible solution
loss mask = assistant completion only
```

不可训练 prompt/system token。

## 7.2 Baseline 2：direct external OPD-PG

Student 当前策略生成：

\[
y \sim \pi_\theta(\cdot\mid x)
\]

Qwen3-8B 用相同 plain prompt 对 student 的**同一组 response token IDs**打分：

\[
r_t^{\text{OPD}} = \operatorname{sg}\left[
\log \pi_{8B}(y_t\mid x,y_{<t})-
\log \pi_{\theta,old}(y_t\mid x,y_{<t})
\right]
\]

MVP 使用 sampled-token policy-gradient OPD（K1），而不是存全词表 logits：

```text
loss = clipped policy-gradient loss
per-token advantage = r_t^OPD
use_task_rewards = false
```

实验名称必须写成：

```text
OPD-PG (sampled-token K1)
```

不能笼统声称是最强 full-vocabulary GKD。

Phase 2 可加：

```text
OPD-topK, teacher top-k=64 or 128, forward-KL/JSD
```

作为更强 direct OPD 确认，不阻塞 MVP。

## 7.3 GRPO control

标准 math-verifier GRPO，所有采样、LoRA 和训练参数与 ExtPI-RLSD 相同，但：

```text
rlsd_lambda = 0
no teacher scorer
```

这是必需对照。没有它无法判断增益来自 PI credit，还是普通 RLVR。

## 7.4 我们的方法：ExtPI-RLSD

Offline PI：

\[
z_{8B} \sim \pi_{8B}(\cdot\mid x),\quad \text{Verify}(z_{8B})=1
\]

Student on-policy trajectory：

\[
y \sim \pi_\theta(\cdot\mid x)
\]

PI scorer 对同一 student token 序列打分：

\[
\Delta_t = \operatorname{sg}\left[
\log \pi_{\bar\theta}(y_t\mid x,z_{8B},y_{<t})-
\log \pi_{\theta,old}(y_t\mid x,y_{<t})
\right]
\]

GRPO 给出 sequence/group-relative advantage `A`，RLSD 构造：

\[
w_t=\exp(\operatorname{sign}(A)\Delta_t)
\]

\[
\tilde w_t=\operatorname{clip}(w_t,1-\epsilon_d,1+\epsilon_d)
\]

\[
A'_t=A\left[(1-\lambda)+\lambda\tilde w_t\right]
\]

然后把 `A'_t` 送入原有 clipped GRPO policy loss。

不新增一个独立 KL loss。环境 reward 决定更新方向，PI 只改变 credit magnitude。

### 7.4.1 PI scorer 模式

代码支持：

```yaml
teacher_update_mode:
  - base_no_adapter
  - current_no_grad
  - periodic_snapshot
```

MVP 默认：

```yaml
teacher_update_mode: base_no_adapter
```

含义：同一个 Qwen3-1.7B PEFT 模型在 scorer pass 中临时 disable LoRA adapter；base weights 冻结，PI 只进入 teacher context。

确认实验追加：

```yaml
teacher_update_mode: periodic_snapshot
teacher_sync_interval: 10
```

### 7.4.2 Shuffled PI control

固定随机 permutation，把别题的 8B trace 配给当前问题。其余配置完全一致。

预期证据链：

```text
ExtPI-RLSD > GRPO
ExtPI-RLSD > shuffled-PI RLSD
```

否则不能把增益归因于 PI 的题目相关语义。

## 8. 单卡 inline scorer 设计

## 8.1 为什么需要它

官方 teacher service/resource pool 通常假设独立 teacher GPU。当前只有 1×H200，因此需要：

```text
student rollout → release/offload rollout KV → inline teacher score → actor update
```

同一时刻只做一种重计算，不要求并行 teacher worker。

## 8.2 统一 scorer contract

```python
@dataclass
class TeacherScoreRequest:
    prompt_input_ids: torch.LongTensor
    prompt_attention_mask: torch.BoolTensor
    response_ids: torch.LongTensor
    response_mask: torch.BoolTensor
    temperature: float
    metadata: dict

@dataclass
class TeacherScoreOutput:
    response_log_probs: torch.FloatTensor  # [B, T_response]
    response_mask: torch.BoolTensor
    metrics: dict

class TeacherScorer(Protocol):
    def score(self, request: TeacherScoreRequest) -> TeacherScoreOutput: ...
```

Backend：

```text
inline_external_hf: Qwen3-8B, frozen BF16
inline_self_peft: Qwen3-1.7B, same actor with adapter disabled or snapshot
```

## 8.3 Exact token alignment

这是最高优先级正确性条件。

禁止：

```python
text = tokenizer.decode(student_response_ids)
teacher_ids = tokenizer(text)   # 禁止
```

必须：

```python
teacher_full_ids = concat(teacher_prefix_ids, original_student_response_ids)
```

因为 teacher prefix 长度可变，response logprob 的 causal shift 要逐样本正确处理：

```text
logits position prefix_len-1 predicts response token 0
logits position prefix_len   predicts response token 1
...
```

MVP scorer 可强制 `micro_batch_size=1`，消除 variable-prefix padding 的首轮风险；通过测试后再向量化。

Teacher 和 student 的 logprob 都按 rollout temperature 计算：

```python
log_probs = log_softmax(logits / temperature, dim=-1)
```

### 8.3.1 Tokenizer startup assertions

Direct OPD 启动时必须验证：

```text
vocab size equal
all token string → id mappings equal
bos/eos/pad ids equal
special-token map equal
chat-template hash equal
```

任何一项不一致，直接失败，不允许文本重 tokenize fallback。

## 8.4 显存与生命周期

External 8B scorer：

```yaml
dtype: bf16
requires_grad: false
eval: true
use_cache: false
attn_implementation: flash_attention_2
micro_batch_size: 1
```

建议在线顺序：

1. 生成 student rollout；
2. 保存 response IDs、old logprobs；
3. 释放 vLLM KV cache/rollout engine 可释放部分；
4. teacher scorer no-grad forward；
5. 计算 reward、advantage、OPD/RLSD signal；
6. actor backward/update；
7. prepare rollout engine 并同步 LoRA。

遇到 OOM 的降级顺序：

1. 降低 vLLM `gpu_memory_utilization`；
2. 降低 `max_num_batched_tokens`；
3. teacher microbatch 保持 1；
4. response length 从 1024 降到 768；
5. 最后才降 online batch。

任何变更必须在所有相应在线方法同步并记录。

## 10. Reward 与 advantage

MVP reward 只使用可验证信号：

```text
correctness: 1.0 if math_verify passes else 0.0
format: optional 0.0–0.05, only for valid boxed format
```

不要用 LLM-as-judge。

GRPO/RLSD 必须记录每个 group 是否有 reward variance：

```text
group_std > 0 ratio
all-correct ratio
all-wrong ratio
```

如果 `group_std > 0 ratio < 0.60`：

1. 先确认 frontier 构建是否正确；
2. 再把 `rollouts_per_prompt` 提到 8；
3. 不要静默训练大量 zero-advantage batch。

## 12. 测试与验收标准

在任何训练前，以下测试必须通过。

### 12.1 RLSD 数学性质

1. `lambda=0` 时，输出 advantage 与 GRPO bitwise/allclose 一致；
2. `teacher_logp == student_logp` 时，所有 weight=1；
3. 对所有非零 `A`，`sign(A') == sign(A)`；
4. reweight 落在 `[1-lambda*eps, 1+lambda*eps]` 对应范围；
5. teacher/student logprobs 均 stop-gradient；
6. response-mask 外 advantage 恒为 0；
7. negative-only 只改变负序列。

### 12.2 Exact alignment

构造不同 prefix 长度的 batch，验证：

- scorer target token IDs 与 student response IDs 完全相等；
- 首个 response token 使用 prefix 最后一位置 logits；
- padding 不进入 logprob；
- decode/re-tokenize 路径不存在；
- temperature 一致。

### 12.3 Prompt privacy

- student prompt serialized bytes 不包含 PI；
- student DataProto/batch 不含 scorer-only PI token tensor；
- checkpoint 不保存原始 API key；
- PI 仅存在于 teacher scorer request。

### 12.4 Objective active

Monkeypatch teacher logprobs：

- 改变 teacher logprob 后，OPD gradient 必须变化；
- 改变 PI teacher logprob 后，RLSD gradient 必须变化；
- 缺少 teacher logprob key 时必须抛错；
- 禁止 silent fallback 到 GRPO。

### 12.5 LoRA 与 teacher

- 只有 LoRA 参数 `requires_grad=True`；
- external 8B teacher 所有 grad 为 None；
- `base_no_adapter` scorer 中 adapter 确实关闭；
- scorer pass 后 active adapter 正确恢复；
- teacher eval mode 不被 actor train mode污染。

### 12.6 运行验收

5-step smoke：

```text
no NaN/Inf
no OOM
reward parser works
teacher metrics nonzero
checkpoint can save and resume
resume 后下一 batch IDs 与未中断 run 一致
```

## 15. 失败诊断决策树

### 15.1 PI gate 不通过

症状：`p_PI ≈ p_plain`。

检查：

1. 8B trace 是否真正正确；
2. PI prompt 是否被 chat template 截断；
3. 1.7B 是否能理解长 trace；
4. trace 是否过长/冗余；
5. non-thinking self-teacher 是否需要短摘要。

修复优先级：

```text
选择最短正确 trace
→ 抽取 8B trace 的关键方法摘要
→ PI max tokens 2048
→ 增加 teacher thinking-mode ablation
```

### 15.2 RLSD ≈ GRPO

检查：

```text
delta 是否接近 0
weight 是否全部被裁剪/全部为 1
group advantage 是否大量为 0
PI prompt 是否误进 student side
teacher scorer 是否实际运行
```

### 15.3 RLSD < GRPO 且长度收缩

依次：

1. 确认 student/scorer style 与 thinking mode一致；
2. 检查高 KL style token；
3. 运行 `negative_only=true`；
4. λ 从 .5 降到 .25；
5. 只在 reasoning span 做 reweight（有稳定 span parser 时）。

### 15.4 Direct OPD 不稳定

症状：logratio p99 极大、clamp ratio 高。

依次：

1. tokenizer/chat template assertions；
2. response offset alignment；
3. teacher/student thinking mode一致；
4. 降 LR；
5. 缩小 logratio clamp；
6. optional top-k support matching。

### 15.5 ExtPI-RLSD 好于 SFT但明显低于 OPD

这是可解释结果，不是代码失败。进一步区分：

- **recipient bottleneck**：1.7B self-teacher 无法把 8B trace转成足够强 token signal；
- **credit bottleneck**：PI uplift存在，但 RLSD weighting过弱；
- **support bottleneck**：student policy 从未采到关键策略。

诊断实验：

```text
1. PI-assisted 1.7B inference upper bound
2. λ ∈ {0.25, 0.5, 0.75}
3. RLSD periodic snapshot
4. 少量 8B trace SFT cold start + RLSD（明确标为额外方法）
```

## 18. 最终主张边界

若实验成功，可以主张：

> Verified reasoning traces from a stronger external model can serve as semantic privileged information for recipient-grounded RLSD, improving the PI-free Qwen3-1.7B policy and approaching sampled-token direct OPD under a single-GPU setup.

不能主张：

- 1.7B 复制了 8B 的完整 policy；
- 这是 direct 8B→1.7B logit distillation；
- 比 full-vocabulary/top-k OPD 更强，除非真正运行该 baseline；
- frontier 结果等同于无偏全分布泛化；
- AIME 小样本差异自动具有统计显著性。

若结果不成功，本设计仍能明确区分：

```text
PI 本身无 uplift
vs self-teacher 无法解释 PI
vs RLSD credit 不够强
vs student on-policy support 不足
vs scorer/token alignment 实现错误
```

这正是快速验证阶段应优先获得的结论。
